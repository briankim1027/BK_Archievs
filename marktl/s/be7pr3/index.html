초록

이 보고서는 Hermes의 brian 프로필 환경을 집의 desktop과 notebook 같은 다른 PC로 옮길 때 필요한 절차와 검증 기준을 정리한다. 핵심은 프로필 자체는 아카이브로 재사용하되, 기기마다 달라지는 경로와 네트워크 값은 새 환경에서 다시 확인하는 것이다.

문제

프로필 파일은 복사할 수 있지만 Obsidian vault 경로, WSL host IP, Ollama 설치 상태, Telegram gateway 운영 주체는 기기마다 달라진다.

해결 방향

export/import 후 .env와 로컬 런타임을 검증한다. 특히 OLLAMA_BASE_URL은 현재 PC 값을 그대로 믿지 않는다.

핵심 발견

1. 프로필은 이식 가능하다

config.yaml, SOUL.md, memories, skills, 기본 프로필 구조는 export/import 방식으로 재사용할 수 있다.

2. 환경값은 재검증해야 한다

Obsidian 경로, Ollama host IP, 모델 다운로드 상태, ffmpeg 설치 여부는 새 PC 기준으로 다시 맞춰야 한다.

3. gateway는 단일 주체가 안전하다

같은 Telegram bot token을 여러 기기에서 동시에 gateway로 운영하면 메시지 처리 충돌 가능성이 있다.

“복사는 프로필까지, 검증은 런타임부터. 이 경계를 지키는 것이 마이그레이션의 핵심이다.”

실행 절차

1. 현재 PC에서 프로필 export

현재 세팅이 끝난 PC에서 brian 프로필을 아카이브로 내보낸다.

hermes profile export brian -o brian-profile.tar.gz

생성된 brian-profile.tar.gz는 OneDrive, USB, NAS, Google Drive, 개인 저장소 등으로 새 기기에 복사한다.

2. 새 PC에서 기본 준비

새 PC에 Hermes를 설치한 뒤 기본 실행을 확인한다.

hermes

현재 작업 흐름은 WSL 기준이므로 가능하면 다음 구성을 맞춘다.

  • Windows + WSL Ubuntu
  • Hermes는 WSL 쪽에서 실행
  • Windows 파일은 /mnt/c/... 경로로 접근

3. 새 PC에서 프로필 import

hermes profile import brian-profile.tar.gz --name brian
hermes profile use brian

이후 프로필 상태를 확인한다.

hermes profile show brian
hermes profile list

기대 결과

brian 프로필이 생성되고 sticky default가 brian으로 설정되어야 한다.

4. 가장 먼저 확인할 파일

hermes -p brian config path
hermes -p brian config env-path
  • ~/.hermes/profiles/brian/config.yaml
  • ~/.hermes/profiles/brian/.env
  • ~/.hermes/profiles/brian/SOUL.md

검증 근거와 설정 포인트

Obsidian 연동

현재 환경의 기준값은 다음과 같다.

OBSIDIAN_VAULT_PATH=/mnt/c/Users/SKTelecom/Documents/Obsidian_DB/Composing
OBSIDIAN_DEFAULT_NOTE_DIR=_Raw

새 PC의 사용자명이나 문서 경로가 다르면 .env를 수정한다.

nano ~/.hermes/profiles/brian/.env
OBSIDIAN_VAULT_PATH=/mnt/c/Users/<사용자명>/Documents/Obsidian_DB/Composing
OBSIDIAN_DEFAULT_NOTE_DIR=_Raw

WSL에서 vault 경로가 존재하는지, _Raw 폴더에 테스트 노트를 생성할 수 있는지 확인한다.

find /mnt/c/Users/<사용자명>/Documents/Obsidian_DB/Composing -maxdepth 2 -type d | head

ffmpeg 설치

TTS와 오디오 변환 안정성을 위해 새 PC에서도 설치를 권장한다.

sudo apt update
sudo apt install -y ffmpeg
ffmpeg -version

Ollama 설치와 모델 준비

Windows 쪽에 Ollama를 설치하고 PowerShell에서 확인한다.

where ollama

현재 사용하는 모델은 gemma4:e4b와 qwen3.5:9b다.

ollama pull gemma4:e4b
ollama pull qwen3.5:9b
ollama list

Windows 환경변수

[Environment]::SetEnvironmentVariable("OLLAMA_CONTEXT_LENGTH","32768","User")
[Environment]::SetEnvironmentVariable("OLLAMA_HOST","0.0.0.0:11434","User")
변수 의미
OLLAMA_CONTEXT_LENGTH=32768 Hermes 같은 tool-using agent에 필요한 충분한 컨텍스트 확보
OLLAMA_HOST=0.0.0.0:11434 WSL에서 Windows Ollama API에 접근 가능하게 함

Ollama 재시작과 리슨 확인

Get-Process ollama -ErrorAction SilentlyContinue | Stop-Process -Force
Start-Process "C:\Users\<사용자명>\AppData\Local\Programs\Ollama\ollama.exe" -ArgumentList "serve"
Get-NetTCPConnection -LocalPort 11434 -State Listen

WSL host IP 확인

새 PC마다 WSL host IP는 달라질 수 있다. 기존 값인 172.25.16.1을 그대로 복사하면 안 될 수 있다.

ip route

출력에서 default via 뒤의 IP를 사용한다.

default via 172.xx.xx.1 dev eth0

OLLAMA_BASE_URL 수정

OLLAMA_BASE_URL=http://<WSL_HOST_IP>:11434/v1

예시는 다음과 같다.

OLLAMA_BASE_URL=http://172.25.16.1:11434/v1

검증 명령은 다음 두 가지다.

curl --noproxy '*' http://<WSL_HOST_IP>:11434/v1/models
curl --noproxy '*' http://<WSL_HOST_IP>:11434/api/tags

모델 전환 구조

현재 brian 프로필에는 기본 모델로 Codex를 쓰고, 추가 custom provider로 Ollama를 붙이는 구조가 들어 있다. import가 정상이라면 config 자체는 대체로 그대로 사용할 수 있다.

Hermes
brian
기본
Codex
custom provider
Ollama
gemma4:e4b
qwen3.5:9b

기본 실행

hermes -p brian

세션 안에서 모델 전환

/model custom:ollama:gemma4:e4b
/model custom:ollama:qwen3.5:9b

Codex로 복귀

/model codex:gpt-5.4

운영 전략

Telegram gateway 주의사항

같은 Telegram bot token은 동시에 한 기기에서만 gateway로 쓰는 것을 권장한다. 여러 Hermes gateway를 같은 토큰으로 동시에 띄우면 메시지 수신과 응답 주체가 꼬일 수 있다.

기기 권장 역할
Desktop 메인 Hermes 운영 머신, Telegram gateway 상시 실행, Ollama/자동화/cron 중심, 무거운 작업 우선 담당
Notebook 이동 중 사용, 문서 작성, Obsidian, 테스트 작업, 필요 시 로컬 모델 사용, gateway는 평소 비활성

gateway 실행 전 확인

hermes -p brian gateway run

충돌이 발생하면 기존 다른 기기의 gateway를 먼저 중지하고 메인 운영 기기 하나만 선택한다.

결론: 같은 환경은 “복사”보다 “재검증”으로 완성된다

Hermes brian 프로필의 이식은 어렵지 않다. 다만 성공 여부는 아카이브 파일 자체가 아니라 새 PC의 실행 환경에 달려 있다. Obsidian 경로, Ollama host IP, 모델 설치 상태, Telegram gateway 주체만 차분히 확인하면 desktop과 notebook 사이에서도 거의 같은 작업 경험을 유지할 수 있다.

실무적으로는 desktop을 상시 운영 중심으로 두고 notebook은 이동성과 문서 작업 중심으로 쓰는 구성이 가장 단순하고 안정적이다.

핵심 문장:
프로필은 옮기고, 경로와 네트워크는 새 기기에서 다시 확인한다.
